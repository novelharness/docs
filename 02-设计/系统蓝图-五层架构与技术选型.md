# 系统蓝图：五层架构与技术选型（面向小说创作平台）

## 1. 你当前思路的判断

你的总体方向是对的，而且是可落地的：

1. **数据与业务能力收敛到 Go+MySQL API 层**（小说实体 CRUD）。
2. **Claude Code 能力拆成“配置中心 + Runtime 执行层”**，避免单体耦合。
3. **编排交给 n8n**，不重复造流程引擎。
4. **前端聚焦创作场景与平台配置**，不把业务逻辑塞进浏览器。

核心优化点不在“方向”，而在**边界和契约**：谁负责状态真相、谁负责任务编排、谁负责模型调用策略。

## 2. 推荐五层架构

```text
L5 前端体验层（Next.js）
  - 小说项目工作台、章节编辑器、流程看板、平台管理

L4 流程编排层（n8n）
  - workflow 模板、任务编排、重试、告警、人工审批节点

L3 智能体执行层（Agent Runtime Service）
  - 多 Agent 协作、上下文构建、记忆检索、工具调用

L2 平台配置中心（Control Plane）
  - MCP/Skill/Agent 模板、模型供应商与路由策略、策略配置

L1 业务数据层（Go API + MySQL）
  - 小说/章节/世界观/伏笔/角色/任务/状态/审计日志 CRUD
```

## 3. 技术选型建议

- **Go（Gin）**：业务 CRUD API、稳定高并发、事务一致性。
- **MySQL**：核心业务主库（强一致 + 事务）。
- **Redis（建议增补）**：会话缓存、短期上下文、任务幂等键。
- **对象存储（建议增补）**：章节版本、长文本快照、导入导出。
- **Node.js 或 Go 做配置中心**：
  - 若目标是复用现有 Claude Code JS/TS 生态，优先 Node。
  - 若目标是统一语言栈、极致运维简化，可选 Go。

## 4. 关键边界（必须先定）

### 4.1 单一事实源

- **小说业务事实源 = Go+MySQL**（章节正文、角色状态、伏笔状态）。
- Runtime 不直接改主库，必须通过业务 API 提交 `state_delta`。

### 4.2 编排与执行分离

- **n8n 负责编排**：顺序、重试、人工审核、超时处理。
- **Runtime 负责执行**：生成、检索、分析、合并。

### 4.3 配置与运行分离

- 配置中心管理模板、版本与策略。
- Runtime 只消费“已发布配置”，避免线上漂移。

## 5. 你方案里最容易踩坑的点

1. **把 n8n 当数据库**：不行，n8n 只做编排，不做业务主存储。
2. **Runtime 直接写 MySQL**：会破坏业务边界，后期难审计。
3. **Agent 配置无版本管理**：改个 prompt 就可能污染所有任务。
4. **模型路由策略散落在各服务**：会导致成本和质量不可控。

## 6. 一句话架构原则

> **Go API 守住业务真相，n8n 编排业务流程，Runtime 负责智能执行，Control Plane 统一配置治理，Next.js 提供创作体验。**
