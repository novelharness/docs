# 常驻智能体运行态设计

## 概述

常驻智能体（Persistent Agents）指在小说创作流程中持续运行、保持自身状态并可随时响应请求的 Agent。与一次性命令式调用不同，它们具有内存、长生命周期和事件循环，能够跨章节、跨卷工作。本文阐述如何设计和部署常驻智能体的运行态。

## 为什么需要常驻智能体？

在大型工程（如百万字小说）中，简单的 stateless 调用无法满足以下需求：

- **上下文共享**：Agent 需要记住之前的决策（如人物关系、伏笔、节奏反馈），避免重复生成或逻辑冲突。
- **长任务分段执行**：有些任务（如写一个长卷）需要多次迭代和反馈循环，常驻 Agent 可以保持状态并连续执行。
- **事件驱动**：Agent 能够监听某些事件（比如读者追读率下降），在需要时自动调整写作策略。

## 运行态架构

常驻智能体运行态可由以下部分构成：

1. **事件循环/调度器**：负责接收任务或事件，并调用内部处理函数。可基于 Node.js（如 `express` + websockets）、Python（如 `FastAPI` + asyncio）构建。
2. **状态存储**：存储 Agent 自身状态，如长记忆摘要、当前任务进度、未完成的 TODO 列表。可存放在内存（用于快速访问）和持久层（文件或数据库）中。
3. **上下文管理器**：读取最近事件与长期记忆，构建发送给 LLM 的上下文。它需要控制上下文大小，通过压缩、摘要或检索的方式确保信息相关且不超出模型窗口。指出 Claude Code 使用三层记忆体系（用户、项目、动态导入）并支持压缩和隔离，这为我们提供了借鉴。
4. **LLM 适配器**：封装与模型交互的接口，处理提示词拼接、工具调用以及响应解析。
5. **技能调度器**：管理外部工具或子任务的调用，例如搜索、数据库查询、随机名生成器等。

## 状态存储与持久化

常驻 Agent 的状态包括：

- **会话历史摘要**：对于长对话或多次调用，需要定期将事件总结成简短摘要以保持可用性。
- **未完成任务列表**：记录当前待执行的子任务、优先级和状态。
- **角色与故事状态**：如人物最新境界、已解锁的金手指等级等。

状态存储可以组合使用内存缓存（如 Redis）与持久化数据库（PostgreSQL、MySQL）。内存缓存用于快速读写，持久化用于在 Agent 重启后恢复状态。使用 cron 任务或在每次状态更新后同步写入数据库。

## 上下文压缩策略

由于 LLM 的上下文窗口有限，常驻 Agent 需要持续压缩和整理记忆：

1. **分层存储**：将长期记忆（世界观、大纲）存入长期存储，只在必要时引入；近期记忆（前几章摘要）保持在短期缓冲区。
2. **内容摘要**：采用摘要模型或手工提炼重要事件、伏笔、角色变化，并用简短自然语言或要点表示。
3. **关键词索引**：为每个章节或事件生成关键词标签，存入向量索引（如使用 `faiss` 或 `pgvector`），在查询时匹配相关事件再展开细节。
4. **定期剪枝**：每完成一个阶段（如每卷）后，对旧记忆进行 `compact` 操作，保留关键信息，删除冗余描述。Claude Code 的 `/compact` 命令即提供类似功能。

## 运行态示例：大纲规划师 Agent

以大纲规划师为例：

- **启动时**加载当前设定库和历史大纲至内存。
- **事件循环**监听“生成新卷”或“调整细纲”事件。
- **上下文管理器**根据任务选择必要的设定和历史卷纲摘要，构造提示发送至 LLM。
- **LLM Adapter**调用模型生成新卷纲，并返回结果。
- **状态存储**更新卷纲，加入未生成章节队列；记录调用次数及反馈。

## 并发与任务队列

为了在多 Agent 协作时避免竞争条件，可为每个 Agent 维护独立的任务队列。例如使用 RabbitMQ、Redis Stream 或数据库表作为队列：

- 每个任务记录输入、状态（未开始/进行中/已完成/失败）、指派的 Agent。
- Agent 拉取自身队列中的任务并标记为进行中，处理结束后更新状态并写入结果。
- 其他 Agent 通过事件或回调机制获得通知，执行下一步。

## 监控与错误处理

常驻 Agent 运行时间长，必须具备健壮的监控：

- **心跳监控**：定时发送心跳信号到监控系统，检测是否崩溃。
- **日志记录**：记录每次 LLM 调用、输入输出、技能调用及错误信息，便于审计。
- **重试机制**：当 LLM 调用失败或返回格式错误时，根据重试策略重新调用或降级到简化逻辑。
- **限流**：控制 Agent 调用 LLM 的频率和并发，防止成本过高或 API 限制。

## 总结

常驻智能体的运行态是一个长生命周期、状态化的服务，拥有事件循环、上下文管理、内存持久化和技能调度等模块。设计时应借鉴 Claude Code 的记忆层次和上下文压缩策略，并结合队列和数据库提供可靠的任务分发与状态恢复能力。这样可以确保在大型小说工程中多 Agent 持续协作而不失控。
