# 初始化流程、回环重写与长篇状态同步方案（n8n + Runtime）

> 这份文档专门回答：
> 1) 初始化世界观/角色/主线如何做成工作流；
> 2) 审核不通过如何 loop 重写；
> 3) 已写 200 万字后如何基于事实继续写且不跑偏；
> 4) 各 Agent 的状态同步到底由谁负责。

## 0. 先给结论（避免方向跑偏）

- **n8n 可以做 loop**，而且很适合做“流程级循环/重试/人工介入”。
- 但“语义级 loop（为什么不通过、怎么改）”应由 **Runtime + Quality Gate** 决策。
- 长篇小说（百万字）不能靠“把全部历史喂给模型”；必须做 **事实层 + 状态层 + 摘要层** 的分层记忆。
- Agent 状态同步不建议靠 n8n 节点内变量硬扛，应该由 **Runtime 持久化状态中心** 统一管理。

---

## 1) 初始化工作流如何设计（世界观/角色/主线）

## 1.1 初始化目标

一次初始化要落地产物（可持久化）：
1. `world_bible`（世界规则、禁忌、力量体系）
2. `character_cards`（主角/配角/阵营关系）
3. `main_arc`（主线阶段目标、里程碑）
4. `constraints`（写作不变量）

## 1.2 n8n 初始化流程（建议节点）

`Webhook(start)`
-> `Runtime.create_project`
-> `Director.bootstrap`
-> `WorldGuardian.validate_world`
-> `CastBuilder.generate_characters`
-> `QualityGate.init_review`
-> IF(PASS?)
   - yes -> `Runtime.publish_baseline`
   - no  -> `Runtime.rewrite_init` -> 回到 `Director.bootstrap`

## 1.3 初始化 loop 的退出条件

- 最大重写次数（如 3）
- 质量分阈值（如 >= 85）
- 人工强制通过/终止开关

> 这些都可以在 n8n 用 IF + Merge + Counter 实现；计数建议落 Runtime，避免流程重启后丢计数。

---

## 2) 审核不通过要重写：n8n 能不能实现？

可以，分两层：

## 2.1 流程层 loop（n8n）

- `IF quality_result == REWRITE`
- 写入 `rewrite_reason` 与 `rewrite_target_agent`
- 跳转到目标节点（例如 protagonist 或 supporting_cast）
- 重跑后再进 `quality_gate`

n8n 擅长这个层面的路由与编排。

## 2.2 语义层 loop（Runtime）

Runtime 要返回结构化原因码，例如：
- `CONTINUITY_BREAK`
- `POWER_SYSTEM_VIOLATION`
- `PACE_COLLAPSE`
- `OUT_OF_CHARACTER`

并附“最小修复指令（patch prompt）”，让被打回 Agent 精确重写，而不是全文重做。

> 关键：n8n 负责“让它再跑一次”，Runtime 负责“告诉它该怎么改”。

## 3) 写到 200 万字后如何不跑偏

## 3.1 三层记忆模型（必须）

1. **事实层 Fact Store（强约束）**
   - 人物事实：身份、关系、能力边界
   - 世界事实：地理、势力、规则
   - 事件事实：关键里程碑、已发生不可逆事件

2. **状态层 Story State（中约束）**
   - 当前卷/章进度
   - 当前冲突态势
   - 角色阶段目标

3. **摘要层 Summaries（弱约束）**
   - 卷摘要、角色弧线摘要、近 N 章摘要
   - 用于降低上下文长度

## 3.2 章节生成时的上下文拼装

每次只拼 4 块：
- `mandatory_facts`（必须遵守）
- `current_state`（当前局势）
- `task_card`（导演下发任务）
- `recent_context`（近几章摘要/片段）

不要把 200 万字直接塞模型，改为检索+组包。

## 3.3 “跑偏防线”

- 生成前：WorldGuardian 预检查（任务是否越界）
- 生成后：QualityGate + ContinuityGuardian 双检
- 入库前：只写增量事实（diff），不全量覆盖

---

## 4) 各 Agent 如何同步状态

## 4.1 不建议方案

- 让每个 Agent 在 n8n 节点里互相传全部状态（会爆）

## 4.2 建议方案（状态中心）

Runtime 维护统一状态中心（DB + Event Log）：
- `runs`：章节运行实例
- `tasks`：agent 子任务
- `facts`：事实表（带版本）
- `state_snapshots`：卷/章快照
- `events`：状态迁移事件

Agent 只拿到“任务所需最小快照 ID”，执行完提交“增量补丁”。

## 4.3 同步协议（最小化）

输入：
- `run_id`
- `task_id`
- `snapshot_id`
- `task_card`

输出：
- `draft_patch`
- `fact_deltas`
- `risk_flags`

由 Runtime 合并，不由 Agent 彼此直接写共享状态。

---

## 5) n8n 与 Runtime 分工边界（你最关心）

- **n8n**：编排、回环、重试、人工审批、通知、可视化。
- **Runtime**：状态机、记忆检索与拼装、质量判定、事实写入、冲突仲裁。

一句话：**n8n 是流程大脑，Runtime 是语义大脑。**

---

## 6) 可直接落地的两条工作流

## 6.1 Workflow-A：项目初始化

`start_project -> bootstrap_world -> generate_cast -> qa_init -> publish_baseline`

带 `REWRITE` 回环与人工审批节点。

## 6.2 Workflow-B：章节生产

`create_run -> director_plan -> parallel_agents -> merge -> qa -> publish`

`qa=REWRITE` 时回到指定 agent；超过阈值转人工。

---

## 7) 下一步实现建议（按优先级）

1. Runtime 增加 `run_id/task_id/snapshot_id` 三个主键语义。
2. 增加 `quality_gate` 结构化错误码 + patch 指令。
3. 上线初始化工作流与章节工作流两条 n8n pipeline。
4. 接入事实库（先 PostgreSQL + JSONB），先做“事实增量写入”。
5. 加一个“状态观测面板”（可先用 n8n + SQL 查询替代前端）。

结论：
- 你提到的 loop、审核打回、长篇状态同步，**n8n 都能参与**；
- 但真正决定“是否跑偏、如何修复”的核心能力必须放在 Runtime。
