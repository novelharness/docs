# Agent 上下文与记忆管理

## 背景

大型小说创作涉及长期且复杂的上下文：世界观设定、人物关系、大纲结构、前文剧情、伏笔与回收等。单个 LLM 的固定上下文窗口无法容纳全部信息。研究指出默认的单 Agent 代码助手在单次提示中往往遗漏远距离依赖。为此，需要通过**上下文工程**和**记忆管理**让每个 Agent 在合适的时刻获取需要的信息，同时保持其长期记忆的一致性。

## 多层记忆体系

Claude Code 引入三层记忆层次：用户级、项目级和动态导入。在小说系统中，可以对应以下层次：

1. **世界级记忆**（User/Project Level）
   - 包含世界观设定、力量体系、金手指说明、人物初始状态等长期不变的信息。
   - 存储在长久的数据库或配置文件中，几乎不需要修改，但需时常读取供参考。
2. **卷级记忆**（Project Level）
   - 包含当前卷的大纲、主线与支线、重要道具和人物状态。
   - 卷结束后将其摘要归档到长期记忆。卷级记忆以结构化 JSON 或文本摘要形式保存。
3. **章节级记忆**（Dynamic Level）
   - 包含最近几章的剧情、未回收的伏笔、读者反馈、节奏分析数据。
   - 用于生成下一章，并通过压缩算法在超出 LLM 上下文窗口前进行摘要或移除。

## 记忆压缩策略

### 摘要与关键点提取

对于长篇文本，可以使用 LLM 生成章节摘要。每个摘要包含以下关键字段：

- **情节回顾**：发生了哪些重要事件。
- **人物变化**：哪些角色升级、受伤、离场或出现。
- **伏笔记录**：埋下了哪些重要伏笔及预期回收位置。
- **节奏标记**：该章属于压抑阶段还是释放阶段，方便节奏分析。

将这些信息以 JSON 形式存入数据库，并只在需要时加载。

### 关键词索引与向量检索

为每个章节生成关键词（可以用 LLM 提取关键名词、事件、地点），并使用向量数据库（如 `pgvector`、`faiss`、`milvus`）将关键词嵌入向量空间。生成章节时，根据当前任务的关键词查询匹配的章节摘要，提取相关信息加入上下文。这样可以在海量文本中快速检索相关内容，而不必载入全部记忆。

### 时间衰减与剪枝

对最近章节赋予较高权重，随着剧情向前推进，逐渐降低旧摘要的优先级。在合适的节点（如一个卷结束后）进行剪枝：将早期记忆合并为长摘要或永久记录，只保留关键设置和伏笔。Claude Code 的 `/compact` 命令提供了将上下文压缩为要点的思路。

### 多 Agent 记忆隔离

不同 Agent 在同一系统中工作时，应该避免上下文冲突：

- **世界观信息只读**：所有 Agent 共享同一份世界设定，但不能修改；修改需通过作者或设定架构师统一处理。
- **任务相关记忆私有**：章节生成器仅接收与当前章有关的摘要，节奏分析师仅需历史节奏记录。
- **共享通道**：通过数据库或消息队列共享必要信息（如节奏分析师的建议）。

这样的隔离可避免 Agent 在无关信息中迷失，同时降低上下文窗口压力。

## 记忆更新流程示例

以下为章节生成任务中记忆更新的推荐流程：

1. **读取世界观与卷级记忆**：从数据库获取基本设定和当前卷的大纲。
2. **检索章节摘要**：通过关键词匹配和向量检索，找到与本章关联性最高的 n 个摘要，拼接进上下文。
3. **生成章节并提取总结**：Story Generator Agent 根据上下文写出章节正文，并在结尾自动生成本章摘要（作为新记忆）。
4. **更新伏笔表**：将新增伏笔写入伏笔追踪表，更新其状态（待回收）。
5. **压缩旧记忆**：根据窗口限制，对最旧且权重最低的摘要执行压缩，生成更短的合并摘要，并删除原始摘要。
6. **同步节奏与状态**：将本章的节奏标记写入节奏记录，并更新角色状态。

## 应用案例：节奏分析师

节奏分析师 Agent 需要回顾前文节奏与本章节奏。它从状态库中提取最近十章的节奏标记和节奏曲线，然后将本章正文分析后标记为“压抑”还是“释放”，并判断是否符合“三章一小爽，十章一大爽”的规律。不符合时给出调整建议，并将分析结果存入新的记忆中。

## 总结

通过构建多层记忆体系、运用摘要、关键词索引和向量检索策略，以及定期进行剪枝和压缩，可以让多个 Agent 在海量背景信息下仍保持清晰的上下文。借鉴 Claude Code 提供的记忆隔离和压缩机制，以及节奏分析对上下文的特定需求，可显著提高小说创作系统的稳定性和质量。
