# n8n + FastGPT + ClaudeCode 协同架构设计（先跑通，再拆分）

> 目标：复用工作流平台能力，最小化自研前端与运维复杂度；把精力集中在「多 Agent 协同创作」与「上下文/记忆治理」。

## 1. 先回答你的核心问题

### 1.1 n8n/FastGPT 里的 Agent 定义，算不算“agent 仓库实现”？

算，但只是**平台内实现**：
- **n8n** 更像“编排层”，节点工作流 + 工具调用 + 路由，不擅长复杂 Agent 资产版本化。
- **FastGPT** 更像“应用层/知识层”，擅长知识库、对话应用、工具接入、模型配置。

它们都能承载 Agent 的一部分能力，但如果你要“主角/配角/导演/世界观守卫”等长期演进，建议保留**独立 Agent 资产库（你现在的 `agents/`）**，平台只做运行时挂载和调度。

### 1.2 Claude Code 能不能定义这些 Agent？

可以。Claude Code 非常适合：
- 维护 `agent.yaml / prompt / output schema` 这类资产；
- 生成和重构 runtime/mcp 代码；
- 自动化回归（契约测试、流程冒烟）。

但 Claude Code 本身不是“7x24 在线运行时”。
所以最优组合是：
- **Claude Code 负责“构建与迭代”**；
- **你自建 Runtime 服务负责“常驻执行”**；
- **n8n/FastGPT 负责“流程编排与可视化运营”**。

---

## 2. 推荐落地架构（单仓 MVP 阶段）

## 2.1 四层职责

1. **Agent 资产层（Git）**
   - `agents/*`: 角色定义（导演/主角/配角/质检等）
   - `skills/*`: 技能包（剧情拆解、连续性检查）

2. **Runtime 执行层（自研服务）**
   - 负责执行状态机：任务领取、上下文裁剪、模型调用、结果校验、回写事件
   - 对外提供统一 API（给 n8n/FastGPT）

3. **Workflow 编排层（n8n）**
   - 负责流程 DAG：启动章节、并行分发、失败重试、人工审核、发布
   - 负责 webhook 入口和运营可视化

4. **应用/知识层（FastGPT，可选）**
   - 负责知识库检索（设定集、角色档案、世界规则）
   - 负责对外对话入口（编辑台、运营台）

## 2.2 为什么这套组合最省事

- 不需要先做前端：n8n + FastGPT 就是天然 UI。
- 把复杂性留在 Runtime：多 Agent 协同策略、记忆治理、质量门禁都在你可控代码里。
- 后续可平滑拆仓：`runtime / mcp / skills` 按边界拆分，接口保持不变。

## 3. 多 Agent 高速协同：关键机制（重点）

## 3.1 协同协议（不是“大家共享全部上下文”）

每个 Agent 只收三类输入：
1. **任务卡（Task Card）**：导演下发目标、约束、产出格式。
2. **最小上下文包（Context Capsule）**：和当前任务强相关的片段。
3. **公共事实快照（Fact Snapshot）**：已确认的世界/角色事实。

这样做避免上下文爆炸，降低 token 成本和冲突率。

## 3.2 导演 Agent 的派发模型

导演不直接写内容，只做：
- 拆任务（主角线、配角线、爽点线、伏笔线）
- 定优先级（P0/P1/P2）
- 设验收标准（字数、冲突强度、连续性约束）

下游 Agent 只对“任务卡”负责，产出后进入合并与质检。

## 3.3 Runtime 内部执行状态机（建议）

`PENDING -> DISPATCHED -> RUNNING -> MERGED -> QA -> PASS/REWRITE`

- `REWRITE` 可带原因码（连续性冲突/节奏断裂/设定越界）
- 自动重试次数由 agent.yaml 控制

## 3.4 记忆分层（必须做）

- **短期记忆**：本章临时上下文（TTL=章节结束）
- **中期记忆**：卷级剧情状态（按卷归档）
- **长期记忆**：世界观事实库（只允许受控写入）

写入长期记忆必须经过 `world_guardian + quality_gate` 双门禁。

---

## 4. n8n 流水线建议（你现在就能跑）

## 4.1 主链路

1. Webhook 接收章节需求
2. 调 Runtime `create_run`
3. Runtime 产出导演任务拆解
4. n8n 并行触发：主角/配角/爽点/守卫
5. Runtime `merge`
6. Runtime `quality_gate`
7. PASS 则入库；REWRITE 则回到指定节点

## 4.2 失败与重试

- n8n 负责网络/超时重试
- Runtime 负责语义失败重试（输出不合 schema、事实冲突）

---

## 5. FastGPT 的位置（可插拔）

FastGPT 推荐用于：
- 检索设定知识库（RAG）
- 对外编辑对话台
- 快速给运营同学提供可视化入口

不建议把复杂协同状态机写死在 FastGPT 应用里；让它做“能力提供者”，Runtime 做“总指挥”。

---

## 6. 仓库结构建议（先单仓，后拆分）

当前单仓建议保留：
- `agents/` 角色资产
- `skills/` 技能资产
- `runtime/` 执行服务
- `mcp/` 工具协议适配
- `n8n/workflows/` 工作流定义

拆分时按顺序：
1. 先拆 `runtime`（最独立）
2. 再拆 `mcp`
3. 最后拆 `skills`（和 agents 视治理方式决定是否同仓）

---

## 7. 两周落地计划（务实版）

### Week 1（跑通）
- Runtime 增加 run_id、状态机、事件日志
- n8n 主链路改为“导演拆解 + 并行子任务 + 质检回环”
- 接入一个事实存储（先 SQLite/Postgres 都可）

### Week 2（提效）
- 引入 Context Capsule 选择器（按任务裁剪上下文）
- 引入输出 schema 校验 + 自动 rewrite
- 加观测：每 Agent 耗时、token、失败原因排行

---

## 8. 你现在最该做的决策（3 选 1）

1. **最省事**：n8n 主编排 + 自研 Runtime（推荐）
2. **最快上线对话台**：FastGPT 前台 + n8n 编排 + Runtime
3. **纯自研**：全自研编排与前端（不建议当前阶段）

结论：你的判断是对的——先复用 n8n/FastGPT，大幅减少前期开发；把核心精力放在 Runtime 协同协议与记忆治理上，项目会更快跑起来。
