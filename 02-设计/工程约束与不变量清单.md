# 工程约束与不变量清单（防跑偏核心）

> 目的：给开发团队“不可违反”的边界，避免实现时方向漂移。

## 1. 系统级不变量（Hard Invariants）

1. **业务真相只在 Go+MySQL**。
   - 禁止 Runtime / n8n 直接写核心业务表。
2. **章节主链路唯一写入口**。
   - 只有 `POST /api/v1/runtime/chapter-commit` 可以提交章节正式状态。
3. **同章节单写者**。
   - 同一 `novel_id + chapter_no` 在同一时刻最多一个 workflow 拥有提交权。
4. **配置可追溯**。
   - 每次执行必须记录 `config_version`。
5. **全链路可追踪**。
   - 每次执行必须携带 `trace_id` 并贯穿前端、n8n、Runtime、Go API。
6. **提交原子性**。
   - 正文、状态、伏笔、审计日志必须同事务成功/失败。

## 2. 安全与权限约束

1. Runtime 使用服务账号，不允许使用用户 token 直写主库。
2. `publish/rollback/chapter-commit` 必须 RBAC + 审计。
3. 模型密钥只存在 Control Plane 或密钥系统，不落地前端。
4. MCP/Skill 仅允许白名单来源，禁止运行未签名技能包。

## 3. 数据约束

1. `chapter_no` 在同小说内唯一。
2. 每次提交必须产生 `chapter_versions` 新版本。
3. `world_dynamic_snapshots.version` 单调递增。
4. 伏笔状态流转只允许：`open -> planned -> paid_off | cancelled`。
5. 任务状态流转只允许按状态机定义，不可跳跃提交。

## 4. 流程约束

1. 未通过自动门禁不得进入提交节点。
2. 强制人工审批章节不得绕过人工节点。
3. 提交失败必须自动回滚并释放锁。
4. 后处理失败不可污染主事务，但必须补偿重试。

## 5. 质量约束（可配置阈值）

1. 一致性分数阈值（默认 >= 85）。
2. 关键冲突数（默认必须 = 0）。
3. 伏笔违规数（默认 <= 1）。
4. 敏感规则命中（默认必须 = 0）。

## 6. 性能与稳定性约束（SLO）

1. 单章主流程 P95 完成时间 <= 8 分钟。
2. 章节提交成功率 >= 99%。
3. 工作流恢复成功率 >= 95%。
4. 重试导致的重复提交率 = 0（靠幂等保障）。

## 7. 观测性约束

1. 每一步节点必须有结构化日志。
2. 关键指标：时延、重试次数、人工介入次数、成本。
3. 每次失败必须有 `error_code`、`error_stage`、`action_hint`。

## 8. 变更约束

1. API 契约变更必须版本化（v1/v2）。
2. Prompt/Agent/Skill 变更必须发布版本并可回滚。
3. 任何不变量变更必须走 ADR 审批。
