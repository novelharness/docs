# 小说工程化任务拆解与流水线设计

## 目标

将“写一部长篇小说”这一宏大任务拆解成多个可管理的小任务，并通过多智能体流水线逐步实现，确保每个阶段都有明确的输入输出和验收标准。本指南在前文 Agent 设计、记忆管理和交互模式的基础上，提供精细化的任务拆解模板和执行流程。

## 总体流程概览

1. **创意确定与需求分析**
   - 作者提供一句话创意和目标受众描述。
   - 设定架构师 Agent 根据创意生成世界观、金手指和角色设定。
2. **高层规划（总纲）**
   - 大纲规划师 Agent 生成起承转合四节点的总纲，明确最终走向和金手指终极形态。
3. **卷级规划**
   - 根据总纲，将故事分割成若干卷，每卷明确核心冲突、主题、出场角色、战力提升、高潮及悬念。
   - 规划卷与卷之间的衔接点（换地图、反派升级等）。
4. **细纲拆解**
   - 将每卷进一步拆解为章节。每章需包含：场景、出场人物、剧情起承转合、爽点、钩子、字数分配。
   - 细纲作为 Story Generator 的施工蓝图，需要精确到每个高潮点位置和伏笔设置。
5. **章节生成与反馈循环**
   - 剧情生成 Agent 按照细纲和上下文写出章节正文及摘要。
   - 节奏分析 Agent 检查情绪曲线，确保符合节奏规律；若不符合，则返回修改建议，让生成 Agent 局部修改。
   - 质量守门员 Agent 检查逻辑、人物一致性、金手指限制，完成最终润色。
6. **角色与支线管理**
   - 建立角色生命周期表，对每个角色的作用阶段、退场节点进行规划。
   - 维护支线任务的闭环，确保支线服务于主线；设计支线支线停留时间不超过 N 章，避免抢戏。
7. **伏笔追踪与回收**
   - 创建伏笔表格，记录伏笔内容、埋设位置、预期回收位置和回收方式。
   - 每完成一章或一卷，检查伏笔表，更新状态或回收准备。
8. **节奏与数据监控**
   - 收集读者追读率、章节留言等数据，输入反馈分析 Agent。
   - 当数据低于阈值（如追读率掉至 40%）时，触发紧急调整机制：可能增加爽点、收束支线或引入重大事件。

## 分阶段任务拆解模板

### 第一阶段：立项与设定生成

**任务 1：核心创意梳理**

- **输入**：作者一句话创意、目标受众描述。
- **输出**：核心卖点（苟道、打脸等）、受众画像。由人工或简单脚本完成。

**任务 2：世界设定生成**

- **Agent**：设定架构师。
- **输入**：核心卖点、受众画像。
- **输出**：世界观设定、金手指说明、角色初始档案、物品/法宝库。
- **标准**：数值化、限制清晰、不自相矛盾。

### 第二阶段：大纲规划

**任务 3：生成总纲**

- **Agent**：大纲规划师。
- **输入**：世界设定。
- **输出**：含起承转合的总纲，明确最终反派与金手指终局。
- **标准**：3000 字以内，总纲含四节点描述和整体概括。

**任务 4：生成卷纲**

- **Agent**：大纲规划师。
- **输入**：总纲、世界设定。
- **输出**：若干卷纲，每卷 10-20 万字，包含核心冲突、出场人物、战力提升、高潮、结尾钩子。
- **标准**：卷与卷之间衔接自然，保留悬念。

**任务 5：生成细纲**

- **Agent**：大纲规划师。
- **输入**：卷纲、世界设定。
- **输出**：章节清单和详细描述，精确到每章 3000-4000 字的节奏分配。
- **标准**：每章都有爽点或悬念，支线不影响主线节奏。

### 第三阶段：章节生成与校验

**任务 6：章节正文生成**

- **Agent**：剧情生成器。
- **输入**：细纲条目、前文摘要、角色状态。
- **输出**：章节正文、章节摘要及角色状态更新。
- **标准**：遵循细纲的起承转合，人物言行符合设定，伏笔适度植入。

**任务 7：节奏分析**

- **Agent**：节奏分析师。
- **输入**：章节正文、节奏历史、预期节奏。
- **输出**：通过/不通过标记，修改建议。
- **标准**：压抑不超过 3 章，释放充分；遵循“三章一小爽，十章一大爽”规则。

**任务 8：质量审校**

- **Agent**：质量守门员。
- **输入**：经调整的章节正文、设定库、角色状态。
- **输出**：最终发布稿、不通过原因及修改建议。
- **标准**：无逻辑漏洞、人物一致性好、语言流畅、不包含毒点。

### 第四阶段：运营与反馈

**任务 9：数据监控与反馈分析**

- **Agent**：反馈分析师（可扩展）。
- **输入**：后台数据（追读率、本章说）、章节发布列表。
- **输出**：调整建议，如增加冲突、节奏调整、角色下线等。
- **标准**：当追读率下降超过阈值时需触发调整；分析建议需要具体可执行。

**任务 10：调整大纲或设定**

- **Agent**：大纲规划师/设定架构师。
- **输入**：反馈分析建议、当前大纲或设定。
- **输出**：更新后的大纲或设定，必要时更新后续细纲。
- **标准**：修改保持一致性，不影响已发布内容；需重新通过节奏和质量检查。

## 任务拆解原则

1. **单一职责**：每个任务只做一件事，明确边界和输出，便于并行化和调试。
2. **可复用性**：拆解后的任务可以在不同项目或不同卷中复用，例如章节生成任务模板可用于所有章节。
3. **可插拔**：任务之间通过标准化接口连接，可插入新的 Agent（例如插入“反派视角生成”Agent）。
4. **渐进式交付**：系统可在每个阶段输出可用成果（如卷纲、章节），允许作者提前阅读并反馈。
5. **反馈驱动**：通过数据监控任务持续收集读者反馈，为下一轮任务提供调整依据。

## 示例：如何拆分“写一卷”任务

假设要完成一个十章的卷，可按照以下子任务执行：

1. 确定本卷主题与核心冲突。
2. 生成卷纲，定义每章概要。
3. 对于每章：
   - 取出章级细纲，调用剧情生成任务生成初稿；
   - 调用节奏分析任务校验并调整；
   - 调用质量审校任务最终发布；
   - 更新记忆并收录伏笔。
4. 完成卷后调用“伏笔回收任务”，将预定在本卷回收的伏笔标记为已回收。
5. 更新卷级记忆摘要，剪枝并压缩旧记忆。

## 总结

通过系统化的任务拆解，小说创作不再是一口气完成功能庞大的任务，而是按照分阶段、分角色的流水线推进。每个任务都有明确的输入输出和验证标准，多 Agent 根据职责协作完成。结合节奏反馈与读者数据驱动，这样的工程化流水线既保证了产出效率，也能保持故事质量和读者吸引力。
