# 智能体设计指南：从静态模板到工程化组件

## 目标

在多智能体小说创作系统中，每个 Agent 负责特定的任务（如世界观设计、大纲规划、章节生成等）。要让这些 Agent 高效协作并可维护，需要将其设计为独立的工程化组件。本文概述了静态层面如何描述一个 Agent，包括提示词模板、配置文件、技能声明和脚本。

## Agent 的静态组成

一个 Agent 不只是一个提示词，它是由**身份定义**、**输入输出规范**和**技能/工具**三个部分组成的集合。在 Claude Code 或类似框架中，一个 Agent 可以使用以下文件描述：

| 文件                   | 作用                                                         |
| ---------------------- | ------------------------------------------------------------ |
| `command.md`           | 定义 Slash 命令名称、系统提示（Agent 身份和任务说明）以及输入参数。实现了 **系统级提示词** 固化，例如描述自己是“大纲规划师，负责生成总纲、卷纲和细纲”。 |
| `prompt.md`            | 可选，用于额外的 prompt 提示，如编写时的格式要求、输出模板等。 |
| `schema.json`          | 描述该 Agent 输入和输出数据的 JSON 模式，方便调用者验证数据结构。 |
| `skills.yaml`          | 声明该 Agent 需要使用的外部技能（如检索工具、随机名生成器、数据库访问等），包括权限和参数。 |
| `examples.md`          | 一系列示例输入与期望输出，用于微调或测试 Agent 行为。        |
| `hook.js` 或 `hook.py` | 钩子脚本，在调用前后对数据进行预处理或后处理，例如验证输入、调整输出格式，或实现强制约束。 |

通过将这些文件存放在版本控制仓库中，可以形成一个标准化的 Agent 模板目录，例如：

```
agents/
├── world_architect/
│   ├── command.md
│   ├── schema.json
│   ├── skills.yaml
│   └── examples.md
├── outline_planner/
│   ├── command.md
│   ├── schema.json
│   └── ...
└── ...
```

这种结构使得 Agent 的行为和输入输出模式完全透明，方便开发者维护和升级。

## 系统提示模板设计

每个 Agent 的系统提示是其核心。一个优秀的提示包含以下内容：

1. **身份定义**：明确角色和任务范围，例如“你是大纲规划师 Agent，负责从设定库生成小说的大纲。”
2. **工作原则**：说明遵守的工程化原则（例如必须遵循细纲格式、避免人物崩坏等）。
3. **输入说明**：描述参数的来源和格式，如“设定库为 JSON 格式，包含世界观、金手指等字段”。
4. **输出要求**：指明输出的结构（如 JSON）和字段含义。
5. **限制与约束**：指出不可违反的设定，例如金手指限制、节奏规律等。

示例（`command.md`）：

```
# /project:outline_planner

你是大纲规划师 Agent，负责根据提供的设定库生成小说的大纲。遵循如下规则：
1. 总纲必须包含起承转合四个节点、金手指终极形态和最终 BOSS。
2. 每卷包含核心冲突、人物列表、主角战力变化、高潮与结尾钩子。
3. 细纲必须细化到每章，列出章节名、场景、出场人物、剧情起承转合、爽点、章末钩子和字数分配。
4. 输出 JSON 对象，字段包括 totalOutline、volumeOutlines（数组）、chapterOutlines（数组）。

输入：设定库 JSON（字段：world, golden_finger, characters,...）。
输出：大纲 JSON。
```

通过系统提示的层层限定，Agent 的输出更加稳定，有助于后续自动处理。

## 输入输出模式（Schema）

定义清晰的输入输出模式有助于调度系统（如 n8n）检查 Agent 返回的数据结构并进行进一步处理。如下是大纲规划师的输出模式示例：

```
{
  "totalOutline": {
    "storyDescription": "...",
    "majorArcs": ["起", "承", "转", "合"],
    "finalBoss": "..."
  },
  "volumeOutlines": [
    {
      "volumeIndex": 1,
      "theme": "矿洞生存", 
      "conflict": "抢夺矿脉",
      "characters": ["主角", "管事"],
      "climax": "反杀管事",
      "hook": "发现上界令牌"
    }
  ],
  "chapterOutlines": [
    {
      "chapterNumber": 1,
      "title": "坠落悬崖",
      "scene": "悬崖边", 
      "characters": ["主角", "师兄"],
      "plot": {
        "rise": "师兄欺辱",
        "climax": "推下悬崖",
        "fall": "金手指觉醒"
      },
      "highlight": "金手指激活展示",
      "hook": "发现尸体" ,
      "wordCount": 3500
    }
  ]
}
```

调度系统可在运行时验证 JSON 结构，如果不符合模式则要求 Agent 重新生成。

## 技能声明与使用

有些 Agent 需要调用外部服务（如检索设定、随机生成人物名称、调用数据库）。在 `skills.yaml` 中声明所需技能，示例：

```
skills:
  - name: search
    description: 使用内置搜索工具检索文档
    parameters:
      query: string
      top_k: integer
  - name: random_name
    description: 从给定文化命名库生成随机姓名
    parameters:
      gender: string
```

运行时，调度系统会加载技能并以 RPC 或 HTTP 调用方式提供给 Agent。

## 钩子脚本（Hook）

对于重要任务，可以在 Agent 调用前后添加钩子脚本。钩子脚本常用于：

- **输入验证**：检查必需字段是否存在，或将文本标准化。
- **输出后处理**：格式化输出、补全缺失字段、加密敏感内容。
- **执行约束**：例如禁止某些词出现，或在金手指限制被突破时报错。

钩子脚本可用 JavaScript/TypeScript（n8n 环境）或 Python（独立服务）编写，并在 `hook.js` 中导出函数供调度器调用。

## 版本控制与测试

建议使用 Git 管理 Agent 模板，以便跟踪修改历史。对于关键 Agent，可以编写单元测试（例如使用 Jest、pytest）验证输出模式和主要逻辑。例如将示例设定输入传入 Agent，然后比对输出 JSON 是否包含必需字段。

## 结论

Agent 设计应遵循模块化、规范化原则，通过系统提示模板、模式定义、技能声明和钩子脚本等静态文件将其行为固定下来。这种工程化的设计使多智能体协作更可靠、可调试和易于扩展，也方便未来迁移到不同的 LLM 框架。
