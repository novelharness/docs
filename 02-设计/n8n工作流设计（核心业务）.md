# n8n 工作流设计（核心业务模块）

## 1. 定位

n8n 是你的**核心编排引擎**，但不是业务数据库、不是模型执行器。

它负责：
- 触发
- 编排
- 重试
- 人工审批
- 告警
- 任务追踪

## 2. 推荐工作流清单（先做 6 条）

## WF-01 创作初始化流

- 输入：题材、世界观草案、角色草案
- 步骤：创建小说项目 -> 初始化静态世界观 -> 初始化角色与伏笔池
- 输出：`novel_id`、初始化报告

## WF-02 章节生产主流程（核心）

- 输入：`novel_id + chapter_no + writing_goal`
- 步骤：
  1. 调 Go API 获取 context bundle
  2. 调 Runtime 生成章节草稿
  3. 调 Runtime 审校
  4. 若失败走修订分支
  5. 通过则调用 Go API commit
- 输出：发布稿或修订建议

## WF-03 伏笔巡检与回收建议流

- 输入：小说范围 + 章节区间
- 步骤：读取 open 伏笔 -> 结合剧情进度分析 -> 给出回收建议
- 输出：伏笔风险报告（超时未回收、密度过高）

## WF-04 世界观一致性巡检流

- 输入：最近 N 章
- 步骤：抽取状态变更 -> 规则校验 -> 生成冲突清单
- 输出：冲突单 + 修复建议

## WF-05 运营反馈驱动优化流

- 输入：追读率、评论标签、掉读点
- 步骤：聚类反馈 -> 映射到节奏/冲突/角色策略 -> 产出调参建议
- 输出：下一章节策略包

## WF-06 模型与成本治理流

- 输入：任务类型 + 成本阈值
- 步骤：模型路由评估 -> 实际成本监控 -> 超阈值告警
- 输出：模型切换建议与成本报告

## 3. 工作流通用规范

1. 每条 workflow 都要有 `trace_id`。
2. 节点失败必须定义：重试次数、退避策略、人工接管点。
3. 所有关键节点写回 `task_runs`。
4. 流程的“最终状态”必须落在 Go API。

## 4. 人工审核节点建议

- 长篇关键转折章
- 新角色首次登场章
- 大高潮前后章
- 风险词命中章

## 5. 性能与稳定性建议

- Runtime 调用设置硬超时（如 120s/节点）。
- Go API 提交幂等，避免重复发布。
- n8n 只传引用和摘要，长文本走对象存储。
